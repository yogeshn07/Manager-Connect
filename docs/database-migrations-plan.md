# Database Migrations Plan

This document specifies the migration strategy, file ordering, sprint alignment, rollback approach, and post-migration steps for the Manager Connect PostgreSQL database. It is an operational reference for the development team. No SQL code is included — each migration is described by what it does, not how it does it.

---

## Migration Philosophy

### Forward-Only Migrations
Manager Connect uses a forward-only migration model. Every schema change is expressed as a new migration file that moves the database forward. There are no rollback scripts. This is the standard approach for production PostgreSQL on Supabase — attempting to reverse-apply a migration after it has been pushed is unsafe because application code and database state must move in lockstep.

### Why No Rollback Scripts
Rollback scripts create a false sense of safety. In practice:
- Application code deployed alongside a migration may depend on the new schema. Rolling back the schema without also rolling back the application binary causes immediate failures.
- Rollback scripts must be written, tested, and maintained in parallel with forward migrations — doubling the maintenance burden.
- Supabase's migration runner does not support transactional DDL rollback in all cases (e.g., `CREATE INDEX CONCURRENTLY` cannot be transactional).

The correct response to a bad migration in staging or production is a **compensating migration** — a new forward migration that reverses the effect.

### One Logical Change Per File
Each migration file contains one cohesive logical change: create one table, enable RLS on one table, add policies for one table, create one function. This constraint makes the migration history readable, makes compensating changes surgical, and keeps each file reviewable without deep context.

### Timestamps Are Supabase CLI-Assigned
Migration file timestamps are generated by the Supabase CLI (`supabase migration new <name>`). The CLI assigns the current UTC timestamp as the prefix. Engineers should never manually edit timestamps or rename migration files after they have been applied to any environment. The ordering of migrations is determined entirely by the timestamp prefix.

## Naming Convention

```
{YYYYMMDDHHMMSS}_{snake_case_description}.sql
```

**Examples:**
- `20240101000001_enable_extensions.sql`
- `20240101000004_create_profiles_table.sql`
- `20240101000005_create_profiles_rls_policies.sql`
- `20240101000006_create_profiles_updated_at_trigger.sql`

**Rules:**
- Table creation and RLS enabling are in the same file (Supabase best practice: enable RLS immediately on creation before any policies are added)
- RLS policy definitions are in a separate file from table creation
- The trigger function definition (`update_updated_at_column`) is one file; attaching it to each table is a separate file per table
- Index creation is included in the same file as the table it belongs to
- Seed data lives in dedicated seed files, never in schema migration files
- Never bundle multiple unrelated tables in one file

## Migration Dependency Order

Some tables reference others via foreign keys. The migration order must respect these dependencies — a table with an FK to another table cannot be created before the referenced table exists.

**Foundational dependency chain:**
1. Extensions and functions must exist before any table
2. `profiles` must exist before any table that references `profiles.id` — which is nearly every table
3. `posts` must exist before `post_images`, `post_reactions`, `comments`, `post_mentions`, `pinned_announcements`
4. `activities` must exist before `activity_rsvps`, `activity_updates`, `polls`, `event_attendance`
5. `polls` must exist before `poll_options`, `poll_votes`
6. `challenges` must exist before `challenge_participants`, `progress_logs`
7. `challenge_participants` must exist before `progress_logs` (progress_logs has an FK to both challenges and challenge_participants)
8. `recognitions` must exist before `recognition_recipients`, `recognition_reactions`
9. `invitations` references `profiles` — must come after `profiles`
10. `flagged_content` references `profiles` — must come after `profiles` (polymorphic `content_id` has no FK, so no dependency on `posts` or `comments`)
11. `pinned_announcements` references both `posts` and `profiles` — must come after both

## Complete Ordered Migration File List

**Note on migration numbering:** The `#` column is a sequential reference label only. Two originally planned migration files were removed (the erroneous `updated_at` trigger migrations for `activity_updates` and `community_health_scores`, neither of which has an `updated_at` column). Four trigger migrations were subsequently added for tables that require `updated_at` (post_reactions, event_attendance, progress_logs, recognition_reactions). The net total is **72 migration files**. Suffix labels (e.g., 16b, 39b) denote insertions into the sequence; actual file timestamps are assigned by the Supabase CLI at creation time.

## Rollback Strategy

### Development Environment
Use `supabase db reset`. This command drops the entire local database and replays all migration files from the beginning. It is fast, safe, and the correct approach for development iteration. Use it freely during Sprint 1–4 development.

### Staging Environment
Create a compensating migration — a new forward migration that reverses the unwanted change. Compensating patterns:

| Scenario | Compensating Migration Action |
|---|---|
| Column added in error | New migration: ALTER TABLE drops the column |
| Wrong CHECK constraint value list | New migration: ALTER TABLE DROP CONSTRAINT, then ADD CONSTRAINT with corrected values |
| RLS policy has wrong condition | New migration: DROP POLICY by name, then recreate with corrected condition |
| Wrong index definition | New migration: DROP INDEX by name, then CREATE INDEX with corrected definition |
| Table created with wrong column type | New migration: ALTER TABLE ALTER COLUMN TYPE (if safe) or create new column, migrate data, drop old column |
| Trigger attached incorrectly | New migration: DROP TRIGGER by name, recreate correctly |

### Production Environment
Never roll back production. Apply a forward-fix migration instead. If the issue is severe enough to block the application, follow this sequence:
1. Deploy a hotfix application release that works with both old and new schema (backward-compatible)
2. Create and test the compensating migration on staging
3. Apply the compensating migration to production during a maintenance window
4. Deploy the final application release that uses the corrected schema

## Post-Migration Steps

After all migrations are applied to a new environment, complete the following steps in order. These steps cannot be expressed as migrations — they require manual action in the Supabase dashboard or via the Supabase Management API.

### 1. Enable Supabase Auth — Phone OTP
In the Supabase dashboard → Authentication → Providers, enable Phone provider and configure the OTP SMS provider (e.g., Twilio). Set OTP expiry to 10 minutes.

### 2. Create Storage Buckets
Create two storage buckets:
- `avatars` — public bucket. Maximum file size: 2 MB. Allowed MIME types: image/jpeg, image/png, image/webp.
- `post-images` — public bucket. Maximum file size: 5 MB per image. Allowed MIME types: image/jpeg, image/png, image/webp.

Set CORS policy on both buckets to allow requests from the Flutter app origin (or configure as open for mobile apps).

### 3. Create the Connect Buddy auth.users Entry
Using the Supabase Auth Admin API (not the dashboard UI — this must be done via the service_role key):
- Create a new user in `auth.users` with a deterministic UUID that will be referenced by the seed migration
- Set the phone number to a non-dialable sentinel value (e.g., `+00000000000`) to prevent OTP login
- Record the UUID — it must be set in the seed migration file before running Phase 9

### 4. Apply Phase 9 Seed Migration
Update `20240101000070_seed_connect_buddy_profile.sql` with the actual Connect Buddy auth UUID from Step 3. Then run `supabase db push` to apply the seed migration.

### 5. Deploy Edge Functions
Deploy all 21 Edge Functions using `supabase functions deploy`. Verify each function's deployment status in the Supabase dashboard → Edge Functions. Functions must be deployed before the app attempts any writes to service_role-only tables.

### 6. Configure Edge Function Secrets
In the Supabase dashboard → Edge Functions → Secrets, set:
- `FCM_SERVER_KEY` — Firebase Cloud Messaging server key for push notification delivery
- Any other third-party API keys used by Edge Functions

### 7. Verify RLS on All Tables
In the Supabase dashboard → Table Editor, confirm that the RLS Enabled badge is shown on all 26 tables. If any table shows RLS as disabled, re-run the corresponding table creation migration (the `supabase db push` command is idempotent for already-applied migrations and will skip them, so the table creation migration cannot be re-run — instead create a compensating migration that enables RLS on the affected table).

### 8. Run Integration Smoke Tests
After all steps above, run the RLS test suite against staging to confirm:
- Active member can read feed posts
- Deactivated member is blocked from all tables
- Admin can see soft-deleted content
- Notification inbox SELECT returns only the recipient's own rows
- Admin audit log SELECT is blocked for members

## Migration Checklist (Per File)

Before pushing any migration to staging or production, verify:

- [ ] Table creation includes `ENABLE ROW LEVEL SECURITY` in the same statement
- [ ] All foreign key references use tables that exist earlier in the migration order
- [ ] CHECK constraint values exactly match the values used in application-layer enums (case-sensitive)
- [ ] Indexes are named descriptively using the pattern `idx_{table}_{column}` or `idx_{table}_{column1}_{column2}`
- [ ] Partial indexes include the WHERE clause in the name for clarity (e.g., `idx_posts_is_deleted_false`)
- [ ] No data mutations in schema migration files (INSERT/UPDATE/DELETE belong in seed files only)
- [ ] The migration has been tested locally via `supabase db reset` followed by `supabase db push`
- [ ] The migration has been applied to the staging environment and integration tests pass
- [ ] No column renames or drops without a prior deprecation migration on staging
- [ ] The migration file name accurately describes the change (someone reading only the filename should understand what it does)
- [ ] If this migration adds a trigger, the trigger function already exists (applied in an earlier migration)
- [ ] If this migration adds an RLS policy, RLS is already enabled on the table (applied in the table creation migration)
